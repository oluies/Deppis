# Phase 1 Data Model: Metadata-Private Messenger

Entities derived from spec.md (Key Entities + FRs). Fields describe meaning, not storage. All
secret-bearing fields are handled per Constitution I/II (vetted libs, constant-time, erased on
epoch rollover). "Public" marks values an observer may see; everything else is confidential.

## User

- `userId` — opaque local identity (never a network-visible long-term identifier).
- `buddies` — set of Buddy relationships, max **512** (FR-015).
- `devices` — set of Device records belonging to this user.
- `schedule` — retrieval schedule parameters the user controls (FR-005).
- Validation: `|buddies| ≤ 512`; adding beyond the limit is rejected predictably.

## Device

- `deviceId` — opaque per-device identity, local.
- `ownerUserId` — the User this device belongs to.
- `epochKeys` — current-epoch per-buddy and device keys; **erased on epoch rollover** (FR-013).
- `providerBinding` — handle to the service-provider buffer this device syncs through (FR-010).
- State: `Online | Offline`. Offline devices catch up via the provider on return (FR-008).
- Invariant: two devices of one user MUST NOT emit linkable/duplicate traffic (FR-011).

## BuddyRelationship

- `pairId` — symmetric, derived from the out-of-band shared secret; identical view for both.
- `selfRole`, `peerRole` — each side is both a sender and a receiver of notifications.
- `safetyNumber` — out-of-band comparison value (FR-001); pairing rejected on mismatch.
- `sendNotificationCredential` — token THIS user holds to notify the peer (issued by the peer).
- `recvNotificationCredential` — token THIS user issued to the peer (receiver→sender direction).
- `ratchetState` — double-ratchet state for content FS with this buddy.
- `tokenCounter` — monotone per-message counter feeding retrieval-token derivation (ensures
  non-recurrence).
- State: `Pending → Confirmed` (after safety-number match) `→ Removed` (FR-018).
- Validation: exactly one Confirmed relationship per pair (FR-002, no duplicates); a Removed
  relationship stops delivery without revealing prior existence to an observer.

## NotificationCredential

- `issuerUserId` (the receiver) / `holderUserId` (the sender).
- `sealedToken` — encrypts a one-hot bit-vector position + aggregation label under the
  notification server's key (D8). Public-on-the-wire as ciphertext only.
- Invariant: lets the holder flip exactly one bit; cannot forge another's bit or exceed its
  allotment (FR-003). Generated by the receiver at add-buddy time, never by the sender.

## Message

- `frame` — fixed **256-byte** content-encrypted frame (FR-015a); size never varies with real
  length. Public: that *a* 256-byte frame exists in a round; Confidential: sender, recipient,
  contents.
- `retrievalToken` — single-use (non-recurrent) token = keyed-PRF(sender id, receiver id,
  counter) (FR-014, D4). Authorizes fetching this one frame without linking fetch to sender.
- State: `Stored (in oblivious store) → Retrieved`. Retention bounded; expiry MUST NOT leak the
  sender (edge case).

## RetrievalToken

- `value` — opaque, single-use. Presenting twice MUST NOT succeed or leak (FR-014).
- Derivation: keyed PRF in `protocol-core`; counter monotonicity guarantees non-recurrence.

## Round

- `roundId` — public, monotone.
- `interval` — public round interval (tuned for minute-order latency).
- Invariant: every participating device emits the same traffic shape each round regardless of
  whether it has a real message (FR-012); carrier/cover frames fill the gap.

## ServiceProvider

- `providerId` — operator identity (mesh, NOT a privacy trust anchor).
- `buffer` — per-device buffered frames + circuit-setup state (atomic transactions).
- `deviceSyncState` — reconciles a user's devices.
- Invariant: full compromise reveals zero buddies and no active/idle determination (FR-010,
  SC-008); untrusted for privacy.

## EnclaveStore (PONG target) — backend, not user data

- `obliviousTable` — stash + bins + message tables, deamortized builds (D7).
- `measurement` — reproducible-build measurement, present in the transparency log (X).
- `attestationKey` — enclave public key, accepted by clients ONLY after a passing appraisal.

## AttestationResult

- `evidence` — SGX DCAP quote / TEE-agnostic token from the enclave.
- `verifierSignature` — Veraison-signed appraisal result.
- `nonce` — client freshness nonce; result rejected if stale (IX).
- `referenceValues` — CoRIM values the appraisal policy pins, from the transparency log (X).
- Invariant: no enclave public key or sealed-token submission before a passing, fresh result.

## BuildPrivacyStatus (cross-cutting — FR-016)

- `backend` — `Dev | EnclaveTarget | GrooveStub | GrooveTarget`.
- `metadataPrivate` — boolean; true ONLY for real backends with passing attestation.
- Invariant: when false, `DEV, NO METADATA PRIVACY` is surfaced in code, logs, and UI; no build
  advertises privacy it does not have.

## Relationships (summary)

```text
User 1───* Device
User 1───* BuddyRelationship *───1 User        (symmetric pair, max 512 per user)
BuddyRelationship 1───1 NotificationCredential(send)  + 1───1 (recv)
BuddyRelationship 1───* Message 1───1 RetrievalToken
Message *───1 Round
User *───1 ServiceProvider          (untrusted)
EnclaveStore 1───1 AttestationResult (gates client trust)
BuildPrivacyStatus  (global, governs labeling)
```
